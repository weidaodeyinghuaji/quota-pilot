# 实时低开销数据刷新设计

## 背景

Codex Quota Glance 的价值依赖状态与 Token 数据足够及时。此前将本地 Codex 检查间隔从 5 秒提高到 10 秒，虽然减少了部分轮询，但牺牲了即时性，而且没有解决详情窗口重复采集、重复 HTTP 请求、重复 JSONL 解析和重复 SQLite 查询等根本开销。

本设计要求在不降低更新频率的前提下降低 CPU、磁盘 IO 和后台进程开销。

## 目标

- 默认及最小 Codex 轮询间隔恢复为 5 秒。
- Codex 会话文件发生变化时立即触发刷新，5 秒轮询作为可靠兜底。
- 胶囊窗口是桌面模式下唯一的数据采集者，详情窗口复用胶囊的数据。
- 同一轮刷新只读取一次活动状态、Token 最新事件和每日汇总。
- 防止慢请求导致下一轮重入和请求堆积。
- 文件监听不可用或异常时自动降级，不影响现有显示和轮询。

## 非目标

- 不改变平台日志同步周期、账户余额缓存周期或用户配置含义。
- 不引入 WebSocket、外部服务或新的运行时依赖。
- 不重构整个本地后端，也不改变现有 SQLite 数据格式。
- 不让设置窗口和更新窗口参与后台采集。

## 方案

采用“事件驱动唤醒 + 5 秒轮询兜底 + 单采集源”的混合方案。

### 后端会话监听

本地后端启动后监听 Codex sessions 根目录。Windows 支持递归目录监听时直接捕获嵌套 JSONL 的新增与修改；监听事件经过短暂防抖后，使会话发现缓存失效，并向 Electron 主进程发出 `codex-session-changed` 通知。

监听仅用于提前唤醒，不作为正确性的唯一来源。监听初始化失败、运行中报错或平台不支持递归监听时，程序继续依赖原有 5 秒轮询。后端停止时必须关闭监听器和防抖定时器。

### Electron 通知链

Electron 主进程订阅后端的会话变化通知，并只向胶囊窗口发送数据失效 IPC。预加载脚本暴露受限的订阅接口，不向渲染进程开放任意 IPC 通道。

胶囊收到通知后立即执行与定时器相同的刷新函数。短时间内连续文件事件会被合并，避免一次写入产生多次刷新。

### 单采集源与详情同步

桌面胶囊继续负责请求本地后端并构建界面状态。详情窗口不再启动 Codex 状态、Token、平台摘要或平台同步定时器，而是通过主进程转发的 IPC 接收胶囊发布的数据状态。

胶囊发布的载荷只包含详情显示需要的可序列化状态：Codex 状态、Token 最新事件、Token 汇总、平台快照、平台本地摘要和错误状态。详情窗口加载完成或打开时主动请求一次最新缓存，避免等待下一次刷新。

非 Electron 的普通浏览器开发模式保持现有独立采集行为，防止开发预览失去数据。

### 合并后端读取

新增一个本地概览接口，在同一请求中返回：

- Codex 账户与活动状态；
- 最新 Token 事件；
- 当日及历史 Token 汇总。

接口内部复用同一轮已经取得的最新 session 路径、文件元数据和增量读取结果。现有独立接口保留，避免破坏已有调用与测试；桌面胶囊切换到概览接口。

### 防重入

刷新协调器遵循以下规则：

- 任一刷新正在执行时，新触发只记录一次 `pending`。
- 当前刷新结束后，如有 `pending`，立即补跑一轮。
- 多个文件事件和定时器同时到达不会并发访问后端。
- 组件卸载后不再更新 React 状态，也不补跑刷新。

这样既不丢失最新变化，也不会因后端偶尔变慢形成请求队列。

## 数据流

1. Codex 写入 session JSONL。
2. 文件监听器使缓存失效并通知 Electron 主进程。
3. 主进程通知胶囊窗口数据已变化。
4. 胶囊刷新协调器立即请求本地概览接口。
5. 后端增量读取新字节、更新 SQLite，并返回统一概览。
6. 胶囊更新自身状态，并将可序列化状态发布给主进程。
7. 主进程缓存最新状态，并转发给已存在的详情窗口。
8. 无文件事件时，5 秒轮询继续执行相同流程作为兜底。

## 错误处理

- 文件监听失败：记录有限诊断信息并继续 5 秒轮询，不反复创建监听器。
- 概览接口部分数据失败：沿用现有可用缓存；返回结构标明不可用部分，不让整个响应失败。
- 详情 IPC 缓存为空：详情显示现有加载/未知状态，主动请求最新缓存，不自行开启第二套轮询。
- Electron 窗口已销毁：主进程发送前检查窗口与 `webContents` 状态。
- 后端关闭：统一释放 watcher、定时器、服务器和数据库资源。

## 测试策略

按 TDD 增加以下回归测试：

- 设置默认值和运行时下限均为 5 秒。
- 会话文件变化会触发通知，并对连续事件防抖。
- 监听失败时后端仍可启动和通过轮询读取数据。
- 概览接口一次返回状态、最新 Token 和汇总，且复用增量读取结果。
- 刷新协调器在并发触发时最多运行一个请求，并在结束后补跑一次。
- Electron 桌面详情窗口不启动独立后台采集。
- 胶囊发布的数据可由详情窗口在加载后立即取得。
- 后端停止后 watcher 和定时器均被释放。

最后运行完整测试、Vite 构建和 `git diff --check`。如条件允许，再以开发包观察空闲与持续写入两种场景下的进程 CPU 和磁盘 IO；性能测量结果单独记录，不以主观感觉替代功能验证。

## 验收标准

- 默认配置下，最迟每 5 秒刷新一次；文件变化通常在一次防抖窗口后触发刷新。
- 打开详情窗口不会增加一套 Codex/平台轮询。
- 慢请求期间不会出现并发刷新堆积。
- 文件监听异常不会导致数据显示中断。
- 原有每日累计 Token、API 今日花费和状态灯行为保持通过。
